# [TIL] 백엔드 5회차 세션 - Spring Boot 핵심 어노테이션, Layered Architecture, REST API 및 명세서 작성

## 1. 스프링의 핵심: 어노테이션 (Annotation)
* **개념**: 사전적 의미로는 '주석'이나, 단순 주석(`// comment`)이 사람만 읽고 컴파일러가 무시하는 것과 달리, **어노테이션은 컴파일러와 프레임워크에게 특별한 의미를 전달하거나 기능을 수행하도록 지시하는 주석**입니다.
* **역사적 배경 (과거 vs 현재)**: 과거의 옛날 Spring 방식에서는 복잡한 XML 설정 파일에 빈(Bean) 등록 코드를 수백 줄씩 수동으로 기술해야 했습니다. 하지만 현재는 클래스나 메서드 위에 어노테이션 포스트잇 하나만 붙여주면 자동으로 등록 및 설정이 완료됩니다.

### 💡 어노테이션의 3가지 주요 역할
1. **표시 (Marker)**: 프레임워크에게 해당 클래스의 역할을 알려줍니다. (예: `@RestController` ➔ "나 컨트롤러야!")
2. **설정 (Configuration)**: 특정 URL 주소와 메서드를 매핑하는 등 웹 설정을 대신합니다. (예: `@GetMapping("/posts")` ➔ "Get/posts 요청을 담당할게")
3. **자동화 (Automation)**: 반복적인 자바 코드를 컴파일 시점에 자동으로 생성해 줍니다. (예: Lombok의 `@RequiredArgsConstructor` ➔ "생성자 자동으로 만들어줘")

## 2. 웹 요청의 내부 동작 흐름 (복습 메커니즘)
브라우저에서 `GET /posts`와 같은 요청을 보냈을 때 스프링 내부에서 일어나는 일꾼들의 순서입니다:
1. **`DispatcherServlet`**: 모든 웹 요청을 가장 먼저 받아내는 총괄 문지기입니다.
2. **`HandlerMapping`**: DispatcherServlet이 "/posts는 누가 담당해?"라고 물어보면 "URL ➔ 컨트롤러 주소록"을 뒤져 매핑된 컨트롤러를 찾아 알려줍니다.
3. **`PostController` (호출)**: 우리가 작성한 코드 영역인 컨트롤러의 메서드(`getPosts()`)가 드디어 실행됩니다.

## 3. 기능별 핵심 어노테이션 총정리

### ① 컴포넌트 스캔 (Component Scan) 관련
어노테이션만 봐도 이 클래스가 스프링 프로젝트 안에서 어떤 역할인지 한눈에 알 수 있으며, 스프링이 이를 인식해 IoC 컨테이너에 **Bean으로 자동 등록**해 줍니다.
* **`@Component`**: 가장 기본이 되는 기반 어노테이션입니다.
* **`@RestController`**: 웹 요청을 받고 JSON/텍스트 데이터를 반환하는 Controller 역할을 지정합니다.
  * 🔍 `@Controller` vs `@RestController` 의 차이:
    * `@Controller`: HTML 파일명(View 이름)을 반환하여 화면을 찾아줌.
    * `@RestController`: 데이터(JSON/문자열)를 브라우저에 그대로 응답함 (`@Controller + @ResponseBody`의 결합 형태).
* **`@Service`**: 비즈니스 로직을 처리하는 계층임을 명시합니다.
* **`@Repository`**: 데이터베이스(DB) 접근을 담당하는 전용 계층임을 명시합니다.

### ② 웹 데이터 파싱 관련 어노테이션 (HTTP 요청 까보기)
클라이언트가 데이터를 어디에 담아 보내느냐에 따라 매핑하는 어노테이션이 완벽히 구분됩니다.

| 어노테이션 | 데이터 추출 위치 | 주요 특징 및 사용 상황 | 예시 코드 스펙 |
| :--- | :--- | :--- | :--- |
| **`@PathVariable`** | **URL 경로 자체**<br>`/posts/1` | `{id}` 경로 변수명과 자바 매개변수명이 동일해야 함. **특정 리소스를 고유하게 식별**할 때 사용. | `Long id` |
| **`@RequestParam`** | **URL 쿼리 스트링**<br>`?page=2&size=10` | URL 물음표(`?`) 뒤에 붙는 파라미터. 데이터 **필터링, 검색, 정렬 옵션, 페이지네이션**에 적합. | `int page`, `String keyword` |
| **`@RequestBody`** | **HTTP 요청 Body**<br>`{ "title": "..." }` | 요청 바디의 **JSON 데이터를 자바 객체로 변환(역직렬화)**해 줌. 주로 데이터 **생성(POST)이나 수정(PUT)**에 사용. | `PostRequest request` |

> **⚠️ `@RequestParam` 유용한 옵션 두 가지**
> * `required = false`: 해당 파라미터가 필수가 아님을 지정 (안 들어오면 null).
> * `defaultValue = "1"`: 파라미터 값이 넘어오지 않을 때 기본값으로 세팅할 데이터 정의.

### ③ 의존성 주입(DI) 및 Lombok
* **`@Autowired`**: 스프링 IoC 컨테이너가 타입을 보고 일치하는 Bean을 찾아 필드에 알아서 주입해 줍니다. (현재는 객체의 불변성을 깨뜨리고 테스트가 어려워 필드 주입 방식인 `@Autowired` 단독 사용은 권장하지 않습니다.)
* **`@RequiredArgsConstructor`**: `final` 키워드가 붙은 필수 필드만을 모아서 **생성자 코드를 컴파일 시점에 진짜로 만들어 넣어주는** 롬복의 핵심 기능입니다. 이 어노테이션 덕분에 가장 안전한 **생성자 주입 방식**을 번거로운 코드 없이 구현할 수 있습니다.
* **Lombok 패키지 세트**: 반복 코드를 줄여주는 개사기 도구들
  * `@Getter`, `@Setter`: Getter/Setter 메서드 자동 완성.
  * `@NoArgsConstructor`: 파라미터가 없는 기본 생성자 자동 생성.
  * `@AllArgsConstructor`: 클래스의 모든 필드를 파라미터로 받는 생성자 생성.

## 4. 스프링 3계층 구조 (Layered Architecture)의 본질

### 🚰 프론트엔드와 백엔드의 아날로그 비유
* **프론트엔드**: 사용자 눈에 보이는 **'수도꼭지(외관)'** 를 편리하고 예쁘게 설계하는 영역입니다.
* **백엔드**: 눈에 보이지 않는 벽 뒤에서 **'물이 길을 따라 잘 나오고 흘러가게 파이프라인(데이터)을 관리'** 하는 영역입니다.

### 🏢 3계층의 확실한 역할 분담 (식당 비유)
컨트롤러 혼자 데이터베이스 연결, 필터링, 정렬, 비즈니스 계산 등 모든 일을 처리하면 코드가 50줄, 100줄 넘게 비대해져 유지보수가 불가능해집니다. 따라서 철저하게 책임을 쪼갭니다.

1. **`Controller` (홀 직원)**: 클라이언트의 요청(주문)을 받고 응답(음식)을 돌려주는 관문입니다. 비즈니스 로직(요리)은 절대 직접 하지 않고 Service에 즉시 위임합니다.
2. **`Service` (주방장)**: 핵심 비즈니스 로직을 담당하는 핵심 계층입니다. **개수 제한 검사, 중복 체크, 정렬, 권한 확인, 판단, 계산, 필터링** 등 '뇌' 역할을 수행합니다.
   * `@Transactional`: 여러 DB 작업(주문 저장 ➔ 재고 감소 ➔ 결제 저장)을 하나의 작업 단위로 묶어, 중간에 하나라도 실패하면 처음 상태로 되돌리는(Rollback) 데이터 정합성 보장 어노테이션입니다.
3. **`Repository` (창고 담당)**: 데이터베이스(DB)에 직접 접근하여 데이터를 저장하고 꺼내오는 일만 전담합니다. 실습에서는 `ArrayList`를 활용해 메모리 위에 가짜 DB를 만들어 데이터를 `add`하고 `findAll`하는 역할을 수행했습니다.

## 5. REST API Architecture 규칙 및 URI 설계

* **개념**: REST(Representational State Transfer) 아키텍처 스타일에 기반한 API로, **자원을 이름(자원의 표현)으로 구분하여 해당 자원의 상태(정보)를 HTTP Method를 통해 명확히 주고받는** 서버 설계 방식입니다. 웹, 모바일 등 다양한 플랫폼과 종속성 없이 원활하게 통신하기 위해 현대 백엔드에서 가장 많이 표준으로 사용합니다.

### 📏 REST URI 설계의 5대 기본 규칙
1. 자원은 명사로 표현하며, 반드시 **소문자**를 사용한다. (대문자 금지)
   * ❌ `/GetTodos`, `/get-todos`
   * ⭕ `/todos`, `/users`
2. URI에 행동을 나타내는 **동사를 절대 넣지 않는다.** 행동은 URL이 아니라 HTTP Method가 표현합니다.
   * ❌ `/createTodo`, `/deleteTodo/3`, `/markTodoAsDone/3`
   * ⭕ `POST /todos`, `DELETE /todos/3`
3. URI 마지막 문자로 슬래시(`/`)를 포함하지 않는다.
   * ❌ `/todos/`
   * ⭕ `/todos`
4. 가독성을 높이기 위해 언더바(`_`) 대신 **하이픈(`-`)**을 적극 사용한다.
   * ❌ `/user_profile`
   * ⭕ `/user-profile`
5. 파일 확장자(`.json`, `.jpg` 등)는 URI 경로에 포함하지 않는다.
   * ❌ `/users/1.json`, `/photo.jpg`
   * ⭕ `/users/1`, `/photos/1`

## 6. API 명세서 작성 항목 구조
스프링 코드 개발에 들어가기 앞서, 프론트엔드 개발자가 "어떤 모양의 데이터가 올지" 알고 화면을 미리 만들고, 백엔드 개발자도 스펙에 맞춰 개발할 수 있도록 **문서를 코딩보다 먼저 작성하는 것이 원칙**입니다.

* **기능**: 이 API가 수행하는 실제 비즈니스 작업 내용 정의
* **Method**: HTTP Method 정의 (`GET`, `POST`, `PUT`, `DELETE` 등)
* **URL**: 자원에 접근하기 위한 엔드포인트 주소 스펙
* **요청 (Request)**: 보내야 할 파라미터 옵션이나 JSON Body의 Key 명세, Nullable 여부
* **응답 (Response)**: 서버가 돌려줄 데이터 구조 및 결과 JSON 텍스트 샘플 예시
* **상태코드**: 정상 성공(`200`, `201`), 클라이언트 요청 오류(`400`), 서버 내부 에러(`500`) 등 반환할 명확한 HTTP 표준 코드 상태 정의
