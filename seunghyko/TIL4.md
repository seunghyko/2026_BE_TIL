# 📝 [TIL] 멋사 제주대 4회차: IoC/DI, Spring Bean, 그리고 Spring MVC 아키텍처

> 💡 **오늘의 핵심 요약**
> "결제 수단 하나 바꿀 때마다 코드를 다 뜯어고쳐야 한다고? -> 스프링이 알아서 조립하게 만들자!"  
> "웹 요청이 들어왔을 때, 매니저(Dispatcher)의 안내를 받아 Controller(점원) - Service(요리사) - Repository(창고)로 역할을 나누고, 데이터는 DTO(쟁반)에 담아서 전달하자."

이번 세션에서는 객체 지향 프로그래밍(OOP)의 다형성을 어떻게 더 잘 활용하는지(IoC/DI/Bean), 그리고 스프링 기반 웹 애플리케이션의 뼈대(MVC)와 내부 동작, 통신 규약(HTTP)에 대해 배웠습니다.

---

## 1. 🚨 기존 코드의 문제점: 내가 다 직접 만들면 생기는 일

객체 안에서 필요한 부품(다른 객체)을 **직접 `new` 키워드로 생성**하면, 나중에 부품을 교체할 때 엄청난 피로감이 발생합니다.

```java
// 기존의 꽉 막힌 OrderService.java
public class OrderService {
    private final PaymentProcessor payment;

    public OrderService() {
        // ❌ 내가 직접 만든다. 새로 결제 수단이 생기면 여기를 직접 고쳐야 함!
        this.payment = new TossPayProcessor(); 
        
        // 만약 이번 분기부터 카카오페이로 갈아탄다면?
        // -> 코드를 매번 수정해야 함 (this.payment = new KakaoPayProcessor();)
    }
}

```

* **문제점:** 결합도가 너무 높아서 결제 수단 하나 바꾸려면 서비스 코드를 직접 열어서 고쳐야 합니다. 다형성을 제대로 활용할 수 없고, 가짜 객체(Mock)를 넣을 수 없어 테스트도 불가능합니다.

## 2. ✨ 구원 투수: IoC, DI, 그리고 Spring Bean (스프링 빈)

이 문제를 해결하기 위해 등장한 것이 **IoC와 DI**, 그리고 **Bean**입니다.

* **DI (의존성 주입, Dependency Injection):** 객체가 자기가 쓸 부품을 직접 만들지 않고, **외부에서 만들어진 부품을 주입받는(넘겨받는) 것**입니다. 덕분에 부품만 슥 갈아끼울 수 있게 되어 결합도가 뚝 떨어집니다.
* **IoC (제어의 역전, Inversion of Control):** "그럼 그 부품은 대체 누가 만들어서 넣어주는데?" -> 바로 **스프링 컨테이너**입니다! 개발자가 하던 객체의 생성과 조립을 스프링이 대신 해주는 것, 즉 **제어권이 스프링으로 넘어간 것**을 제어의 역전이라고 합니다.
* ☕ **Spring Bean (스프링 빈):** 스프링 컨테이너가 직접 생성하고 관리하는 부품(객체)들을 말합니다.
* 개발자가 클래스 위에 `@Component`, `@Controller`, `@Service` 같은 어노테이션(포스트잇)을 붙여두면, 스프링이 "아! 이건 내가 관리해야 할 Bean이구나!" 하고 알아서 컨테이너에 등록하고 필요한 곳에 알아서 조립(DI)해 줍니다.



### 🔧 DI(의존성 주입)의 3가지 방식: 왜 '생성자 주입'을 써야 할까?

스프링이 Bean을 조립해 줄 때 사용하는 3가지 방법이 있습니다.

1. **필드 주입 (Field Injection):** 변수 위에 `@Autowired`를 달아버리는 방식. 편하지만 테스트가 힘들고 스프링에 너무 종속적이라 **현재는 비추천**합니다.
2. **수정자 주입 (Setter Injection):** `set~` 메서드를 통해 주입하는 방식. 나중에 부품을 유연하게 바꿀 수 있지만, 실수로 누군가 바꿀 위험이 있어 잘 쓰지 않습니다.
3. **🏆 생성자 주입 (Constructor Injection):** 객체가 생성될 때 딱 한 번만 주입받는 방식. **현재 스프링에서 가장 강력하게 권장하는 표준 방식**입니다. 한 번 조립되면 바뀔 일이 없어 안전하며, 롬복(Lombok)의 `@RequiredArgsConstructor`와 함께 쓰면 코드가 마법처럼 깔끔해집니다.

## 3. 🏗️ Spring MVC 구조와 동작 원리

"요청을 **역할**로 나누는 구조"   
웹 프로그램이 커지면 모든 로직을 한 곳에 몰아넣을 수 없습니다. 그래서 햄버거 가게처럼 **요청을 역할로 나누는 구조**를 가집니다. 이 과정에는 우리가 만든 계층뿐만 아니라 입구에서 손님을 안내하는 스프링 내부의 숨은 일꾼들이 함께 일합니다.

### 🕵️‍♂️ 스프링의 숨은 일꾼들 (요청이 Controller에 오기까지)

* **DispatcherServlet (매장 총괄 매니저):** 사용자의 모든 HTTP 요청을 가장 앞단에서 먼저 받아내는 문지기입니다. 요청이 들어오면 직접 처리하지 않고 "이거 누가 처리해야 해?" 하고 안내 데스크에 물어본 뒤 적절히 배분하는 지휘관 역할을 합니다.
* **HandlerMapping (안내 데스크 / 배차표):** 총괄 매니저가 "이 `/posts` GET 요청 누구한테 줘야 해?"라고 물어보면, 등록된 주소록을 뒤져서 "아, 그건 `PostController` 담당입니다!"라고 정확히 찾아내어 연결해 줍니다.

### 🍔 4가지 핵심 계층 (우리가 직접 만드는 부분)

숨은 일꾼들의 안내를 받은 요청은 본격적으로 아래 4개의 계층을 거쳐 처리됩니다.

1. **Controller (점원):** 사용자(클라이언트)의 요청을 가장 먼저 받고, 응답을 돌려주는 역할. 손님에게 주문을 받고, 어떤 요리사(Service)에게 일을 시킬지 결정한 뒤, 요리가 완성되면 다시 손님에게 서빙합니다.
2. **Service (요리사):** 실제 핵심 **비즈니스 로직**을 처리하는 역할. 점원에게 주문서를 넘겨받아 가격을 계산하거나 할인 쿠폰을 적용하는 등 복잡한 규칙이나 계산을 수행합니다.
3. **Repository (창고 관리자):** DB(데이터베이스)에서 데이터를 꺼내오거나 저장하는 역할. 요리사가 데이터(고기)를 요구하면 순수하게 DB(냉장고)와 소통하여 데이터를 찾아다 줍니다.
4. **DTO (Data Transfer Object, 서빙 쟁반 / 포장 박스):** **계층 간 데이터를 주고받을 때 사용하는 객체(바구니)**. DB에서 갓 꺼낸 날것의 엔티티(Entity)를 손님에게 그대로 던져줄 수 없기 때문에, 클라이언트에게 딱 필요한 데이터만 예쁘게 가공해서 담아 전달하는 전용 상자입니다.

## 4. 🌐 통신 규칙: HTTP Method (요청의 목적)

스프링 컨트롤러에서는 클라이언트가 **어떤 목적**으로 요청을 보냈는지에 따라 어노테이션으로 매핑합니다.

* **GET (조회):** 데이터를 달라고 요청할 때 (`@GetMapping`)
* **POST (생성):** 새로운 데이터를 저장해달라고 요청할 때 (`@PostMapping`)
* **PUT/PATCH (수정):** 기존 데이터를 수정해달라고 할 때
* **DELETE (삭제):** 데이터를 지워달라고 할 때

### 💻 Controller & DTO 코드 예시

```java
@RestController // 나는 점원(Controller Bean)이야!
public class PostController {

    // 게시글 작성 (POST)
    @PostMapping("/posts")
    public String writePost(@RequestBody PostCreateDto dto) {        
        // @RequestBody: 손님이 넘겨준 데이터를 DTO(전달 상자)에 담아서 받아옴
        return "작성 완료: " + dto.getTitle();
    }
}

```

---

**💡 복습 포인트:**
객체를 직접 `new`로 만들지 않고 '생성자 주입'을 통해 스프링에게 제어권을 넘기면(IoC/DI), 스프링이 알아서 **Bean**으로 등록하고 조립해 줍니다. 웹 요청이 들어오면 `DispatcherServlet`과 `HandlerMapping`의 안내를 받아 **Controller - Service - Repository** 계층으로 철저하게 분업화되어 로직이 처리되며, 이들 사이에서 데이터를 나를 때는 **DTO**라는 안전한 상자를 이용한다는 것을 명심하자.
